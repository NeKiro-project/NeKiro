本模块为纯文档仓库，不包含可执行代码，采用「一特性一目录」的扁平结构：每个 `NNN-<feature-name>/` 子目录是一个独立特性的完整规格包，内部固定包含以下文件以统一交付物形态——
- `spec.md`：主规格，承载用户故事、需求（FR-xxx）、关键实体、成功标准（SC-xxx）、假设与非目标；
- `contracts/`：该特性对外暴露的接口契约（如 `catalog-api.md`、`workspace-installation-api.md`、`runtime-contract.md`），是后续 Go 映射与校验器的输入；
- `data-model.md`：领域数据模型定义；
- `plan.md` / `tasks.md`：实现计划与任务拆解；
- `quickstart.md` / `research.md`：快速上手与技术调研；
- `checklists/requirements.md`：需求核对清单。

编号从 001 到 019，体现演进顺序：001 奠定 Invocation/A2A/Card 语言中性契约基础，随后 002~008 依次补齐 Catalog 注册发现、Workspace 安装约束、Agent Pin 选择、能力解析授权、安装检查与生命周期，009 验收证据、010~012 路由与调度、019 嵌套调用 SDK。各特性通过 `contracts/` 中的 API 文档形成单向依赖方向：上层特性（如安装、路由）引用下层特性（如 Catalog、Invocation 契约）的 contracts 文件，而非反向。
