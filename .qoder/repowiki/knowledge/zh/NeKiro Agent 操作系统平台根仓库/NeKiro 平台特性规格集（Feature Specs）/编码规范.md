- 每个特性目录使用三位数字前缀加连字符命名（`001-complete-invocation-contracts`），并按功能域递增编号，体现平台能力的分阶段交付顺序。
- 所有特性遵循统一的目录骨架：`spec.md` + `contracts/` + `data-model.md` + `plan.md` + `quickstart.md` + `research.md` + `tasks.md` + `checklists/requirements.md`，不得省略任一文件。
- 需求条目使用 `FR-00x` 编号并在 `spec.md` 中以「MUST/MUST NOT」句式书写，成功标准使用 `SC-00x` 编号并给出可量化阈值（如 100%、≤1s）。
- 每个 User Story 必须附带「Why this priority」「Independent Test」和至少一条 Given/When/Then 形式的 Acceptance Scenarios，作为测试与 Review 的可追踪输入。
- `contracts/` 下的契约文件以 `<domain>-api.md` 或 `<domain>-contract.md` 命名，仅描述语言中立的请求/响应/错误语义，不绑定具体 HTTP 路径或框架注解。
- 每个特性末尾明确列出 Assumptions 与 Non-Goals，并将跨运行时证明（Cross-runtime proof）作为 Runtime/Platform Boundary 的子项写入，用于区分 NeKiro 平台职责与外部 Agent Runtime 职责。
